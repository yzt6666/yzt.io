K8S控制器之Statefulset

Statefulset 控制器：概念、原理解读

StatefulSet 是 Kubernetes 中的一种控制器（Controller），用于管理有状态应用程序的部署。与Deployment 控制器不同，StatefulSet 为每个 Pod 实例分配了一个唯一的标识符，并确保这些标识符在 Pod 重新创建时保持不变。这为有状态应用程序提供了一些关键的功能和保证，例如稳定的网络标识符、有序的部署和扩展以及持久化存储。

StatefulSet 的一些基本介绍

  1. 唯一标识符：每个 StatefulSet 管理的 Pod 实例都被分配了一个唯一的标识符，通常是一个整数索引。这个标识符可以用于在网络中唯一地标识每个 Pod 实例。
  2. 稳定的网络标识符：StatefulSet 中的每个 Pod 实例都有一个稳定的网络标识符，通常是一个DNS 名称。这使得其他应用程序可以通过名称轻松地访问特定的 Pod 实例，而不需要关注其具体的 IP 地址变化。
  3. 有序的部署和扩展：StatefulSet 控制器确保 Pod 实例按照定义的顺序逐个启动和关闭。这对于依赖先前实例状态的应用程序非常重要。此外，扩展 StatefulSet 时，新的 Pod 实例也会按照指定的顺序逐个创建。
  4. 持久化存储：StatefulSet 允许每个 Pod 实例关联一个持久化卷（Persistent Volume），这使得有状态应用程序可以在 Pod 重新创建时保留其数据。这为应用程序提供了持久化存储的能力，使得数 据不会丢失或重置。

Statefulset 资源清单文件示例

# 定义一个Kubernetes Service资源  
apiVersion: v1 # 使用Kubernetes API的版本1  
kind: Service # 资源的类型为Service  
metadata:  
  name: nginx # Service的名称是nginx  
  labels: # 为Service添加的标签  
    app: nginx # 标签的键为app，值为nginx  
spec:  
  ports: # 定义Service的端口  
  - port: 80 # Service暴露的端口号为80  
    name: web # 端口的名字为web  
  clusterIP: None # 设置clusterIP为None，表示这是一个Headless Service，不会分配一个集群IP地址  
  selector: # 使用标签选择器来选择Pods  
    app: nginx # 选择标签为app=nginx的Pods  
  
# 分隔符，用于分隔多个Kubernetes资源定义  
---  
  
# 定义一个Kubernetes StatefulSet资源  
apiVersion: apps/v1 # 使用Kubernetes Apps API的版本1  
kind: StatefulSet # 资源的类型为StatefulSet  
metadata:  
  name: web # StatefulSet的名称是web  
spec:  
  selector: # 使用标签选择器来选择Pods  
    matchLabels: # 匹配具有指定标签的Pods  
      app: nginx # Pods必须带有标签app=nginx  
  serviceName: "nginx" # 关联的Service名称，用于Pod的网络发现和负载均衡  
  replicas: 2 # StatefulSet创建的Pod副本数量为2  
  template: # 定义Pod模板，用于创建StatefulSet中的Pods  
    metadata:  
      labels: # Pods的标签  
        app: nginx # Pods的标签，与selector匹配  
    spec:  
      containers: # 定义容器列表  
      - name: nginx # 容器的名称是nginx  
        image: nginx # 容器使用的镜像名称是nginx  
        imagePullPolicy: IfNotPresent # 镜像拉取策略，如果本地有则不拉取  
        ports: # 定义容器暴露的端口  
        - containerPort: 80 # 容器内部监听的端口号为80  
          name: web # 端口的名字为web

headless service作用

Headless Service的主要作用是为分布式系统中的应用提供稳定的网络标识和持久化存储关联。在k8s中，当创建一个Headless Service时，Kube-dns或CoreDNS会为服务下的每个Pod创建唯一的DNS A记录（对于IPv4）或AAAA记录（对于IPv6）。这使得客户端可以直接通过Pod的hostname或子域名访问到具体的Pod，而无需通过负载均衡器。这种方式允许应用程序直接与后端Pod进行通信，非常适合于有状态应用，如数据库集群、消息队列集群等。
