Pod的核心概念

pod的工作方式

在k8s中，所有的资源都可以用一个yaml文件来创建，创建pod也可以使用yaml配置文件。

pod最基础写法（pod-tomcat.yaml）：

apiVersion: v1  #api版本
kind: Pod  #资源类型
metadata:  #pod的元数据
  name: tomcat-test  #pod的名称
  namespace: default #pod的命名空间
  labels:   #pod的标签
    app: tomcat  #定义的标签，键值对
spec:  #开始定义pod的规格，包括容器、存储、网络
  containers:  #容器
  - name: tomcat-java  #容器的名称
    ports:  #容器的监听端口
    - containerPort: 8080  #指定容器应该在其内部监听8080端口
    image: tomcat-8.5-jre8:v1  #使用的Docker镜像
    imagePullPolicy: IfNotPresent  #镜像拉取规则

创建pod：

kubectl apply -f pod-tomcat.yaml

查看pod：

kubectl get pod

kubectl get pod -o wide

查看pod日志：

kubectl logs tomcat-test

删除pod：

kubectl delete pods tomcat-test

进入pod：

kubectl exec -it tomcat-test - - /bin/bash

假如pod里有多个容器，进入指定容器：

kubectl exec -it tomcat-test -c tomcat-first – /bin/bash

查看pod的详细信息：

kubectl describe pod tomcat-test

直接定义一个pod资源，pod被删除就会彻底删除，不会再创建一个新的pod，所以需要一个控制器来管理。

常见的管理pod的控制器：Replicaset、Deployment、Job、CronJob。

deployment管理pod

将镜像上传到node1节点

docker load -i nginx.tar.gz

配置deployment清单文件（nginx-deploy.yaml）

apiVersion: apps/v1
kind: Deployment
metadata:  # 定义Deployment的元数据
  name: nginx-test  # Deployment的名字是nginx-test
  labels:   # 为Deployment添加标签。
    app: nginx-deploy  # 标签的键是app，值是nginx-deploy。
spec:  # 定义Deployment的具体规格。
  selector:  # 用于选择哪些Pod属于这个Deployment。
    matchLabels:  # 通过标签匹配来选择Pod。
      app: nginx  # 选择的Pod必须具有app=nginx的标签。
  replicas: 2  # Deployment需要确保有两个Pod实例在运行。 
  template:    # 定义用于创建新Pod的模板
    metadata:   # 定义Pod模板的元数据。 
      labels:  # 为Pod模板添加标签。
        app: nginx  # Pod模板的标签键是app，值是nginx。
    spec:  # 定义Pod模板的具体规格。 
      containers:   # 定义Pod中要运行的容器列表。
      - name: my-nginx   # 定义Pod中要运行的容器列表。
        image: nginx:v1  # 容器使用的镜像是nginx:v1。 
        imagePullPolicy: IfNotPresent  #容器的拉取策略。
        ports:  # 定义容器要暴露的端口列表。
        - containerPort: 80  # 容器暴露的端口是80。

更新资源清单文件

kubectl apply -f nginx-deploy.yaml

会发现启动了两个，删除其中一个，deploy就会再创建一个，一直保持两个的状态。

根据标签查看pod

kubectl get pod -l app=nginx

查看deploy

kubectl get deploy

查看rs

kubectl get rs -l app=nginx

deploy是rs的升级版，底层是rs。

创建pod的流程

第一步：

客户端提交创建Pod的请求，可以通过调用API Server的Rest API接口，也可以通过kubectl命令行工具。如kubectl apply -f filename.yaml（资源清单文件）

第二步：

API Server接收到Pod创建请求后，会将yaml中的属性信息（metadata）写入etcd。

第三步：

API Server触发watch机制准备创建Pod，信息转发给scheduler，调度器使用调度算法选择node，调度器将node信息给API Server，API Server将绑定的node信息写入etcd。

第四步：

API Server又通过watch机制，调用kubelet，指定pod信息，调用Docker API创建并启动pod内的容器。

第五步：

创建完成后反馈给kubelet，kubelet又将pod的状态信息给API Server，API Server又将pod的状态信息写入etcd。

镜像的拉取策略

  • Always：不管本地是否存在镜像，都要重新拉取镜像。
  • Never：从不拉取镜像。
  • IfNotPresent：如果本地存在，使用本地镜像，本地不存在，从官方拉取镜像。
