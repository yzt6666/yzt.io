个人介绍

面试官您好，我叫杨子腾，毕业于保定学院，就读于软件工程专业，在校期间，我深入学习了Linux系统、数据库原理、Docker等核心课程，对运维产生了浓厚的兴趣，大三的时候参加学校组织的实习工作，在某某公司担任运维工作，主要是维护公司的服务器集群，在此期间，接触到了运维的整体工作流程，对各项工作有了充分认知，也开始自主学习k8s、nginx、tomcat等相关技术，实习六个月后回到学校，自己又找了实习单位，xx公司，在公司里我参与了多个项目的部署和维护工作，比如，我负责搭建并且维护了k8s集群、实现了nginx高可用集群配置、参与了数据库集群的搭建等，积累了丰富的实战经验，我的自我介绍结束了，非常感谢您给我这个机会来介绍自己，我期待能够进一步了解贵公司和这个职位。

你们公司的服务器架构是什么样的？

我们公司现在用的是微服务架构，总共大概有50多台服务器呢。其中，两台Master掌控系统的核心运作，有两台Nginx负责网站的流量分配，三台Tomcat用来处理业务逻辑，还有一台专门的数据库服务器，确保数据的安全和高效访问。这些就构成了我们公司的web网站，给大家提供了稳定又流畅的服务。除了这些，其他的服务器都是按照甲方的要求，用来进行软件的开发和测试工作。这样，我们不仅能满足公司的日常运营需求，还能确保软件开发和测试工作顺利进行。

你在工作中遇到过什么问题？

在我初入第一家公司的时候，主要负责服务器的日常维护工作，有次服务器报警，我迅速定位问题，发现存日志的磁盘占用已经超过设置的阈值，我立刻向上报汇报，请求急购新的磁盘，在得到公司配备更大磁盘容量后，我直接将其安装在服务器上，然后挂载在存储日志的目录，当检查的时候，发现以前的日志没有了，我以为是新磁盘的挂载覆盖掉了旧数据，就很慌，然后我就上csdn查找怎么恢复数据，后来发现是日志并没有被覆盖，还在旧磁盘上，我又重新挂载旧磁盘，将数据拿到其他磁盘，再转移到新磁盘才解决问题，经过这次经历，我意识到数据备份的重要性，以防数据丢失。

你曾经做过什么项目？

我在上一家公司参与过k8s的部署工作，我记得当时根据甲方公司的业务需求，用了3台Master和20多台node，每台服务器都先安装ntpdate来同步时间，然后安装docker和containerd以及所需的依赖包，为了快速部署k8s，我们选择用kubeadm来初始化集群，设置三台Master，配合Nginx和Keepalived实现高可用，确保集群能稳定运行，然后将node节点加入到集群当中，最后在所有节点安装上Calico，来实现集群内部的网络通信。创建了几个pod测试成功后就投入使用了。

这个项目中你有遇到什么问题吗？

我记得是没有什么大问题，就是在测试的时候，停掉主master，发现另外两个master节点上都有vip，也就是发生了脑裂，因为是最近部署的，问题肯定出在keepalived的配置文件上，检查的时候发现两台从节点的优先级是一样的，复制过来没有改才造成这个问题，修改完成后重启服务，vip才正常漂移。

pod的创建流程

首先是客户端提交创建pod的请求，调用api server，api server接收到pod创建请求后，会将编辑好的yaml中的信息写入etcd，然后api server触发watch机制准备创建pod，信息转发给调度器scheduler，调度器使用调度算法选择匹配的node，将node信息反馈给api server将绑定的node信息写入etcd，api server又通过watch机制，调用kubectl，指定pod信息，调用docker api创建并启动pod内的容器，创建完成后反馈给kubectl，再传给api server，将pod的状态信息写入etcd。

如何实现nginx的高可用？

首先要在每台nginx服务器安装Keepalived，Keepalived可以实时监控nginx的运行状态，一旦发现nginx宕机，它就会采取行动，先配置Keepalived，主要是两部分，一个是虚拟IP地址，也叫vip，这是客户端请求的目标地址，还要定义好每个nginx的优先级和权重，来决定在发生故障时，vip漂移到哪个备份服务器。还要确保主从nginx服务器的配置和数据的一致性，最后都启动keepalived服务，它就可以监控nginx服务，从而实现高可用性。

你都创建过什么镜像？

我创建过nginx镜像，首先是创建一个工作目录，将所需要的静态资源，初始镜像，配置文件放到目录里，开始编写dockerfile文件，先导入官网的nginx基础镜像包，再将我修改过的nginx配置文件替换掉原有的文件，把静态资源复制进去，暴露Nginx的80端口，再加上启动nginx服务的命令，文件就写完了，执行docker build命令，将自动创建镜像，完成后再设置好标签，上传到harbor仓库储存备用。大概就是这么一个过程。

pod为什么pending？

可能是集群中没有足够的资源来调度pod（如CPU、内存、存储等），Pod可能由于调度约束（如亲和性/反亲和性规则，污点和容忍）而无法被调度到任何可用的节点上。可能是镜像拉取失败（如拉取的是国外镜像源）。

pod创建失败该怎么办？

pod创建失败可以先查看一下Pod日志，看看失败的原因，可能是镜像不存在或损坏、配置错误、资源限制、网络连接等等，如果是镜像的原因，就检查并修复镜像，或者重新拉取镜像，如果是配置错误，就检查pod的配置文件，修复可能存在的错误，如果是资源限制，就为pod设置适当的资源请求和限制，查看网络策略，看是否禁止创建pod。

简述deployment和replicaset的区别

Deployment 是一个高级别的控制器，管理 ReplicaSet 并提供滚动更新和回滚等高级功能，而 ReplicaSet 是一个低级别的控制器，确保在集群中始终运行指定数量的 Pod 副本。Deployment 适用于需要高级控制和自动化的复杂应用程序，而 ReplicaSet 适用于需要简单、可扩展且灵活的机制来管理 Pod 副本的场景。

就绪探针与存活探针区别是什么？

两者作用不一样，存活探针是将检查失败的容器杀死，创建新的启动容器来保持pod正常工作；
就绪探针是，当就绪探针检查失败，并不重启容器，而是将pod移出endpoint，就绪探针确保了service中的pod都是可用的，确保客户端只与正常的pod交互并且客户端永远不会知道系统存在问题。

copy和add的区别

copy只能从构建上下文中拷贝文件和目录，add除了上述功能，还具有解压缩和自动处理远程文件的功能。

nginx有哪些模块，你用过哪些模块实现过什么功能？

Nginx是一个功能强大的高性能HTTP和反向代理服务器，Nginx的模块可以分为核心模块、标准HTTP模块、可选HTTP模块、邮件服务模块、Stream服务模块和第三方模块。我用过rewrite模块进行url重定向，proxy模块作为反向代理，将客户端请求转发到后端服务器，使用gzip模块将响应的内容进行gzip压缩，减少网络传输的数据量，提高了传输效率，upstream模块配置了后端服务器地址、端口，实现了负载均衡等一些模块。

nginx脚本

#!/bin/bash
#判断nginx是否存活

nginx=$(ps -ef | grep nginx | wc -l)

if [ “${nginx}” -gt 1 ]; then
echo “nginx正常运行中”
else
echo “nginx挂掉了，正在准备重启”
systemctl start nginx.service
sleep 2
status=$(systemctl status nginx.service | grep active | awk ‘{print $2}’)
if [ “${status}” == active ]; then
echo “nginx已经重启成功”
else
echo “nginx重启失败，请手动检查”
fi
fi

__END__
